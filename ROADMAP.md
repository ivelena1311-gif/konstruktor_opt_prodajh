# Дорожная карта разработки приложения
## AI-сервис «Конструктор оптовых продаж»

---

## 1. Резюме проекта

AI-сервис «Конструктор оптовых продаж» — веб-приложение, которое помогает fashion-брендам продавать коллекции оптовым покупателям (байерам) быстрее и по более высокой цене. Пользователь за 4 шага заполняет данные об изделии, и система автоматически генерирует продающую оптовую карточку, PDF-презентацию коллекции, скрипты переговоров и расчёт экономики для магазина-партнёра.

Ключевая боль целевых клиентов — производителей одежды с оптовым каналом — это отсутствие профессиональных материалов для работы с байерами: нет визуально сильной карточки, нет аргументов по марже, нет системы касаний после выставки. Сервис закрывает все эти пробелы за один сеанс работы.

На выходе клиент получает комплект материалов, готовых к немедленному использованию: загрузить на маркетплейс, отправить в WhatsApp байеру, распечатать на выставке или разослать по базе магазинов. Продукт позиционируется не как «AI для карточек», а как **«инструмент закрытия оптовых сделок»**.

## 2. Область работ и границы (Scope)

### Что входит в MVP (обязательно)

- 4-шаговый мастер ввода данных (Wizard): Изделие+Фото → Цены+Партия → Рынок+Покупатель → Усиление продаж
- Все поля необязательные; кнопка «Заполнить примером» на каждом шаге; прогресс-бар «Шаг X из 4»
- Загрузка фотографий изделия (до 6 штук: спереди, сзади, детали) и предпросмотр
- Генерация текстового контента через Claude API: 5–7 B2B-преимуществ, блок «Почему коллекция продаётся», аргументы для закупщика
- Скрипты: звонок закупщику, сообщение в WhatsApp, follow-up после выставки
- Блок расчёта экономики: маржа %, прибыль с партии, точка безубыточности (работает при частичных данных)
- Сборка карточки: компоновка загруженных фото + блоки УТП + цифры экономики (без AI-перерисовки фото)
- Экран результата с 4 табами: Карточка / Презентация / Скрипты / Экономика
- Экспорт: скачать PDF, кнопки «Скопировать текст / WhatsApp / Скрипт звонка / Follow-up»
- Режим «Быстрый старт»: при минимуме данных — 3 уточняющих вопроса и результат всё равно выдаётся
- Деплой на Vercel + страница с примером результата и демо-режим

### Что НЕ входит в MVP (отложено на V2)

- AI-фотосессия: очистка фона, lifestyle-кадры, генерация изображений (Leonardo/Stability/Replicate) — нестабильность результата критична для fashion
- История проектов: сохранение и повторная выгрузка PDF (требует авторизации и базы данных)
- Кнопка «Усилить карточку» (перегенерация)
- Блок «Вероятность продажи» (AI-анализ риска нереализации партии)
- Полноценная авторизация / личный кабинет / управление подпиской
- Интеграции с CRM, маркетплейсами

### Основные допущения

- Разработка ведётся вайб-кодингом (Lovable + Claude Code), без найма backend-разработчика
- Команда: 1–2 человека на старте
- AI-обработка фото в MVP — только компоновка и оформление загруженных снимков, не AI-генерация
- Целевой рынок: РФ и СНГ, fashion-бренды с оптовым каналом, MOQ от 100 единиц
- Монетизация запускается параллельно с MVP: оплата за карточку или подписка

## 3. Архитектура и дизайн на высоком уровне

### Ключевые компоненты

| Компонент | Роль |
|---|---|
| Frontend (SPA) | 4-шаговый Wizard + экран результата с табами + кнопки экспорта |
| Backend / BFF | Приём данных и фото, оркестрация запросов к Claude API и PDF-генератору |
| Claude API | Генерация всего текстового контента (claude-sonnet-4-6) |
| PDF-генератор | HTML-шаблон → конвертация (html2pdf.js или серверный Puppeteer) |
| Файловое хранилище | Временное хранение загруженных фото (Cloudinary / S3 / base64) |
| Хостинг | Vercel (frontend + serverless functions) |

### Потоки данных

```
Пользователь заполняет Wizard
         ↓
Клик «Сгенерировать» → данные + фото → Backend
         ↓
Backend → Claude API (промпт с данными изделия)
         ↓
Claude API → текстовый контент (JSON)
         ↓
Backend собирает данные + тексты → HTML-шаблон → PDF
         ↓
Результат → Frontend → 4 таба → Скачать / Скопировать

При нехватке данных: Backend → режим fallback (черновики, плейсхолдеры)
```

### Основные сущности данных

```
Изделие:
  - тип (dropdown), название, фото[] (до 6), состав, плотность,
    размерная сетка, особенности[]

Цены:
  - оптовая цена, РРЦ, себестоимость, MOQ, срок поставки

Рынок:
  - канал продаж, сегмент (mass/middle/middle+/premium),
    география, сезон, ЦА

Акценты:
  - выбранные УТП (до 2 из списка), главная боль байера,
    требуемые выходы (чекбоксы)

Результат:
  - тексты (JSON), PDF blob/URL, скрипты (строки)
```

## 4. Рекомендуемый стек (2 варианта)

### Вариант A — Быстрый MVP

| Слой | Решение |
|---|---|
| Frontend | Lovable или Next.js (React) |
| AI | Claude API (claude-sonnet-4-6) через Anthropic SDK |
| PDF | html2pdf.js или jsPDF (клиентская библиотека) |
| Фото | Cloudinary free tier или base64 в памяти |
| Хостинг | Vercel (деплой одной командой) |

**Плюсы:** запуск за 3–5 дней, нет сложной инфраструктуры, бесплатный хостинг
**Минусы:** клиентский PDF ограничен по качеству, нет персистентности
**Риски:** качество PDF в браузере варьируется, Cloudinary free tier ограничен

### Вариант B — Масштабируемый production

| Слой | Решение |
|---|---|
| Frontend | Next.js 14+ (App Router), TypeScript, Tailwind CSS |
| Backend | Node.js (Fastify) или serverless-функции |
| AI | Claude API + версионирование промптов |
| PDF | Puppeteer (headless Chromium на сервере) |
| База данных | Supabase (PostgreSQL + Storage + Auth) |
| Хостинг | Vercel (frontend) + Railway/Render (backend) |

**Плюсы:** высококачественный PDF, история проектов, аналитика, подписка — всё достижимо
**Минусы:** 2–3 недели до первого релиза; нужна базовая DevOps-экспертиза
**Риски:** Puppeteer нестабилен на serverless; стоимость Railway ~$5–20/мес

> **Рекомендация:** начать с Варианта A для быстрой валидации гипотезы. Мигрировать на Вариант B после первых 10–20 платящих клиентов.

## 5. План по этапам (милстоуны)

### Этап 0 — Discovery / Прояснение требований `[S, ~2–3 дня]`

**Цель:** зафиксировать финальный scope MVP, выбрать стек, закрыть открытые вопросы до начала кода.

**Deliverables:**
- Финальный список экранов и полей Wizard
- Визуальный референс карточки-образца (скриншот / PDF примера)
- Черновики промптов для Claude API
- HTML-прототип шаблона карточки
- Зарегистрированные аккаунты (Anthropic, Cloudinary/Supabase, Vercel)

**Задачи:**
- [ ] Сделать 10+ тестовых запросов к Claude API с разными наборами данных изделий — оценить качество
- [ ] Нарисовать wireframes 4 шагов Wizard + экрана результата
- [ ] Сделать HTML-прототип карточки вручную по образцу
- [ ] Протестировать PDF-генерацию на прототипе в Chrome и Safari
- [ ] Зафиксировать ответы на открытые вопросы (см. раздел 9)

**Definition of Done:**
- Есть утверждённый образец карточки — визуальный референс
- Claude API даёт качественные тексты на 5 из 5 тестовых запросов
- PDF из HTML рендерится корректно в Chrome и Safari

## 7. План качества и тестирования

### Что тестируем

- **Unit-тесты:** логика калькулятора экономики (маржа%, точка безубыточности), парсинг ответа Claude API, сборка промпта из данных формы
- **Integration-тесты:** полный сценарий «заполнить форму → получить результат» с моком Claude API; PDF-генерация с тестовыми данными
- **E2E-тесты** (Playwright / Cypress):
  - демо «Заполнить примером» → экран результата → скачать PDF
  - пустая форма → режим «Быстрый старт» → результат
- **Ручное тестирование** перед релизом: кросс-браузер (Chrome, Safari, Firefox), мобильный вид, PDF на разных устройствах

### Минимальные требования к CI/CD

```
GitHub Actions:
  - ESLint + Prettier → блокируют merge при нарушениях
  - Unit-тесты → блокируют merge при падении

Vercel:
  - Preview deployment для каждого PR
  - Production deployment при пуше в main

Секреты:
  - Claude API key только через Vercel Environment Variables
  - Никогда не в коде и не в git-истории
```

### Релизные гейты (перед запуском в production)

- [ ] Все unit-тесты зелёные
- [ ] E2E «Заполнить примером → скачать PDF» работает в Chrome и Safari
- [ ] Время от «Сгенерировать» до результата < 30 секунд
- [ ] PDF открывается и читается на мобильном и десктопе
- [ ] Платёж проходит в тестовом режиме (Stripe test mode)
- [ ] Нет P0-багов: приложение не падает при пустых полях или ошибке Claude API

## 8. Безопасность и данные

### Аутентификация и авторизация

- **MVP:** авторизация не обязательна. Доступ ограничивается счётчиком генераций по session/cookie
- **V2:** Supabase Auth или Auth0 (email/пароль + Google OAuth)
  - Роли: `user` / `admin`
  - Администратор: просмотр аналитики, управление тарифами

### Защита данных

- Все запросы через HTTPS (Vercel обеспечивает автоматически)
- Claude API key — только в серверных env-переменных, никогда не передаётся на клиент
- Загруженные фото: хранение только в рамках сессии (до 24 часов), затем автоудаление
- Rate limiting на API-эндпоинтах: не более N запросов в час с одного IP
- Бизнес-данные клиентов (цены, себестоимость) — не логировать в общих логах

### Логи и аудит

| Что логировать | Что не логировать |
|---|---|
| Количество генераций | Содержимое форм |
| Ошибки Claude API (без данных) | Фотографии изделий |
| Время ответа на запрос | Бизнес-данные клиентов |

- Алерт при >5% ошибок Claude API → Telegram-бот или email

### Чеклист по приватности

- [ ] Что собираем: email (при auth), параметры формы — только для генерации результата
- [ ] Зачем: исключительно для генерации карточки, не для рекламы
- [ ] Срок хранения фото: только время сессии, удаляем после генерации PDF
- [ ] Оферта и политика конфиденциальности: разместить **до** публичного запуска (обязательно по 152-ФЗ)
- [ ] Согласие на обработку данных: чекбокс при первом использовании

## 9. Открытые вопросы

Вопросы, которые напрямую влияют на проектирование и разработку:

1. **Образец карточки.** В документе упоминается «как в твоём примере», но сам пример не приложен. Нужен конкретный визуальный референс (скриншот/PDF) для разработки HTML-шаблона.

2. **Стартовый тариф.** Подписка требует Auth, «оплата за карточку» — счётчика, «Коллекция под ключ» — уточнения: это автоматическая генерация или ручная услуга? Нужно выбрать один приоритетный вариант для MVP.

3. **Авторизация в MVP?** Без неё — запуск быстрее, но нет истории и подписки. С ней — +5–7 дней разработки. Ответ определяет объём и архитектуру.

4. **Платёжный шлюз.** Stripe (международные карты) или ЮKassa/Tinkoff (рублёвые платежи РФ)? Влияет на выбор и сроки интеграции.

5. **Допустимое время генерации.** Claude API отвечает 10–25 сек, PDF-рендер ещё 5–10 сек. Нужен ли прогресс-индикатор или асинхронная обработка с уведомлением?

6. **Хранение черновиков.** В localStorage (пропадают при смене устройства) или серверное хранение? Документ упоминает «автосохранение черновика», но не уточняет способ.

7. **Структура PDF-презентации.** Названы только 5 из 10 слайдов (концепт, позиционирование, портрет покупателя, прогноз спроса, расчёт прибыли). Нужно описание оставшихся 5 для разработки шаблона.

8. **Приоритет устройства.** Пользователи будут заполнять с телефона или с компьютера? Влияет на UX chips-выборов, upload фото и превью карточки.

9. **Пакетный режим.** Документ описывает одно изделие как вход, но говорит о «коллекции» как единице продукта. Нужен ли batch-режим (несколько изделий в одном PDF) в MVP?

10. **Водяная метка / брендинг на PDF.** Выходные файлы брендируются логотипом сервиса? Это инструмент виральности (клиент отправляет байеру PDF — байер видит название сервиса).
